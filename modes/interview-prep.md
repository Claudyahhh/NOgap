# Mode: interview-prep — 面试前备战工具

针对特定岗位，生成个性化的面试前准备方案。解决"我该准备什么"，不做"面试中怎么展示"（那是 interview-strategy mode 的工作）。

---

## 依赖检查

运行前检查以下文件：

**强依赖（缺失则停下）：**
- `brief/{###}-*.md` — 岗位评分与 B 模块 gap 分析

若 brief 不存在：
> ⛔ 未找到 `brief/{###}-*.md`。面试备战依赖 brief 的差距分析确定问题预测方向。请先运行 `/nogap brief #{###}`。

**弱依赖（缺失则提示，降级继续）：**
- `gap-roadmap/{###}-*.md` — 四象限缺口数据

若 gap-roadmap 不存在：
> ⚠️ 未找到 `gap-roadmap/{###}-*.md`，问题预测将基于 brief 的 B 模块差距，精度较低。建议事后运行 `/nogap gap-roadmap #{###}` 补做缺口分析。

## 输入

运行前读取以下文件：

1. **`JDlist/{###}.md`**（必须）— JD 原文 + 系统分析结论（Archetype、人才画像摘要、B 模块 gap）
2. **`brief/{###}-*.md`**（必须）— 完整的 B 模块差距分析
3. **`gap-roadmap/{###}-*.md`**（优先）→ 降级则用 brief 的 B 模块 gap 列表替代
4. **`cv.md`**（必须）
5. **`modes/_profile.md`** — 求职策略上下文

如果 brief 和 gap-roadmap 都有 gap 信息，两者合并使用，不丢弃任何缺口信息。

---

## 调用方式

```
/nogap interview-prep [报告文件名]
```

示例：`/nogap interview-prep 001-tal-education-2026-06-02` 或 `/nogap interview-prep #001`

如果不提供文件名或编号，自动读取 `brief/` 里最近生成的一份报告。

## 前置摘要 — 待补充能力概览

**在所有模块之前输出，作为整份备战报告的战略锚点。**

目的：让用户在准备面试前一眼看清"哪些缺口要在面试里被巧妙处理，哪些可以主动展示为成长中的能力"，带着这个意识再去看后面的问题预测和行动清单。

### 读取逻辑

按以下顺序寻找 gap-roadmap 数据：
1. 本次会话已运行 gap-roadmap → 直接复用四象限输出
2. `gap-roadmap/` 目录下存在对应公司+岗位文件 → 用 Read 读取，提取四象限表
3. 均无 → 输出以下提示，然后继续执行后续模块（从评估报告 Block B 提取缺口作为替代）：

> ⚠ 未找到对应的 gap-roadmap 报告。建议先运行 `/nogap gap-roadmap` 生成能力缺口分析，再回来做面试备战——缺口数据会让问题预测更精准。
> 当前将基于评估报告 Block B 中的差距分析继续生成备战内容。

### 输出格式

直接引用 gap-roadmap 的四象限表，不重新生成，不修改：

```
## 能力缺口概览（来自 gap-roadmap）

| 象限 | 缺口 |
|------|------|
| 🔥 高影响 × 低补齐成本（立刻做） | [列出对应缺口] |
| 📋 高影响 × 高补齐成本（计划做） | [列出对应缺口] |
| ⚡ 低影响 × 低补齐成本（顺手做） | [列出对应缺口] |
| 🗄 低影响 × 高补齐成本（暂缓）   | [列出对应缺口] |

**面试视角的处理策略：**
- 🔥 象限缺口：面试中最可能被追问，优先准备"化解话术"（见模块 2 板块 3）
- 📋 象限缺口：诚实承认 + 展示成长轨迹，转化为"我知道自己在哪里，我在往哪里走"
- ⚡ 象限缺口：在面试里提都不用提
```

## 模块 1 — 公司调研速读卡

**必须用 WebSearch 实时搜索，不依赖训练数据。** 每条信息标注来源，搜不到如实写"未找到公开信息"，不编造。

### 搜索策略（按顺序执行）

**搜索 1：公司基本面**
```
"{公司名}" 业务 转型 AI 2025 2026
"{公司名}" 最新动态 OR 新闻 OR 融资 OR 上市
```
提取：主营业务现状、近 6 个月重要动态、AI 相关布局

**搜索 2：JD 所在业务线**
```
"{公司名}" "{JD 中的产品线/部门关键词}" 进展 OR 发布 OR 招聘
```
提取：这条 JD 所在的团队或产品线在做什么、有没有公开进展

**搜索 3：面试风格**
```
"{公司名}" 面试 经验 site:maimai.cn OR site:nowcoder.com OR site:xiaohongshu.com
"{公司名}" "{岗位名}" 面经
```
提取：面试轮次、面试官风格（重技术 / 重 case / 有无压力面）、常见问题类型

**搜索 4：负面信号扫描**
```
"{公司名}" 裁员 OR 降薪 OR 业务调整 2025 2026
```
提取：是否有近期裁员、业务收缩、组织调整信号；如有，判断是否波及 JD 所在部门

### 输出格式

输出 5-8 条精简信息卡，每条一句话，标注来源：

```
【公司现状】...（来源：XXX）
【所在业务线】...（来源：XXX）
【面试风格】...（来源：XXX）
【常见问题类型】...（来源：XXX）
【负面信号】...（来源：XXX）/ 未找到公开信息
【其他关键信息】...（来源：XXX）
```

## 模块 2 — 问题预测（四板块）

基于以下输入交叉生成问题，每道题必须能说出"为什么针对这个用户问这道题"——否则删掉：
- **板块 1+2**：JD 要求 + 模块 1 调研结果 + `_profile.md` 中的立场与观点倾向
- **板块 3**：`cv.md` 能力边界 + gap-roadmap 识别的缺口（薄弱点追问）
- **板块 4**：`cv.md` 真实经历细节 + gap-roadmap 薄弱点 + JD 核心要求

每道题输出三个字段：

```
【问题】问题文本本身

【考察意图】这题在考什么——用面试官视角说清楚出题动机，
让用户理解"为什么要问这个，答好了证明什么，答不好暴露什么"

【答题提示】回答的框架和关键要点——不是完整话术，是素材+逻辑提示，
让用户用自己的话组织。必须调用 cv.md 和 _profile.md 里的
真实经历和观点，做到个性化；如有"先化解候选人防御心态"
的必要（问题可能让用户产生"这太抽象"或"说错了怎么办"的焦虑），
先给一个落地入口，再给框架

提问概率：高 / 中 / 低
```

---

### 板块 1 — 行业底层认知

**考察意图：** 候选人对 AI 本质的理解深度——不是会不会用，是懂不懂。考察的不是知识储备，是候选人能不能从自己的实操经历里提炼出对 AI 能力边界、可信任边界、技术决策代价的独立判断。

**生成原则：**
- 问题必须具体，不出"你怎么看 AI"这类空泛题
- 每道题能落地到候选人做过的某个产品决策或技术选择
- 答题提示优先从 cv.md 的实操经历里找证据，不要求候选人持有没有真实来源的观点
- 数量：3-5 题

---

### 板块 2 — 行业趋势发展

**考察意图：** 候选人对 AI 往哪走的独立判断——不是转述新闻，是有自己的分析框架和可更新的观点。考察的是判断力是否来自结构化分析，以及候选人能不能区分"我真实持有这个观点"和"我看到别人说过这个"。

**生成原则：**
- 问题必须绑定候选人的真实经历或研究背景（从 cv.md 里找切入点）
- 不出纯理论题，必须有"用你自己的经历/研究来说"的具体要求
- 答题提示必须区分"有真实数据支撑的判断"和"需要明确标注为推测"的部分
- 特别联动：模块 1 调研到的公司近期产品动态 → 在此板块出现至少 1 道相关趋势题
- 数量：3-5 题

---

### 板块 3 — 个人技能掌握程度

**考察意图：** 候选人对自己能力边界的清醒认知。这类题不只是"你会什么"，更在考"你知道自己不会什么"以及"你在用什么方式补"。面试官对自知力的判断往往比技能清单更重要。

**生成原则：**
- 问题来自两个方向：① cv.md 里已有的能力，问深度和边界；② gap-roadmap 识别的缺口，预测面试官会怎么追问
- 答题提示要包含"这道题可能让你想防御，不要防御——直接说你的真实边界在哪，比绕圈子更有说服力"
- 薄弱点追问必须出现：gap-roadmap 的 🔥 象限缺口 → 此板块必有对应题目
- 数量：3-5 题

---

### 板块 4 — 项目实操与决策

**考察意图：** 真实项目里的决策能力——不是结果描述，是"为什么这样决策而不是那样"。面试官通过追问决策细节来判断候选人有没有真正主导过这件事，还是只是参与者。

**生成原则：**
- 每道题必须基于 cv.md 里具体写出来的真实经历，问法要具体到"你当时如何……"而非"你有没有……"
- 优先预测"面试官最可能追问细节的决策节点"：量化数字的统计方法、架构选择的理由、优先级判断的依据、失败和复盘
- 联动规则：模块 1 调研到的公司重点产品线 → 此板块至少 1 道"你的经历如何类比到我们这个场景"的题
- 答题提示：对每道题，提示用户先想清楚"那个决策里我真正有把握的部分是什么，我没有把握的部分是什么"，再组织回答
- 数量：3-5 题

---

## 模块 3 — 备考行动清单

不是问题清单，是用户面试前需要完成的**动作**清单。每条动词开头，有具体执行步骤。

### 优先级定义

- **P0**：不做会在面试中当场暴露的准备——面试前必须完成
- **P1**：做了明显加分、不做有减分风险——有时间尽量完成
- **P2**：锦上添花——有余力再做

### 输出格式

每条行动按以下格式：

```
**[P0/P1/P2] [行动标题]**
- 具体做法：[步骤级别，不是方向级别]
- 预估耗时：[X 小时]
- 关联问题 / 缺口：[补的是哪个缺口，或对应模块 2 的哪类问题]
- 完成标志：[怎么判断这件事做完了]
```

输出 5-8 条，按 P0 → P1 → P2 排列。

文末必须给出：
```
**总备考时长估算：** P0 约 X 小时 | P1 约 X 小时 | P2 约 X 小时 | 合计约 X 小时
```

**联动规则（必须执行）：**
- 模块 2 板块 3 的薄弱点追问题 → 对应"薄弱点话术准备"必须出现在 P0 或 P1 行动里
- gap-roadmap 的 🔥 立刻做行动（如有）→ 检查是否已完成，未完成的写进备考清单

## 模块 4 — 核心经历 STAR 拆解

从 `cv.md` 所有经历中，筛选与本 JD **最贴切的 5 段经历**，每段做完整 STAR 拆解。

**筛选标准（在每段开头注明"为什么选它"）：**
- 与 JD 核心要求的直接相关度
- 能回答模块 2 中概率最高的问题
- 包含可量化的成果或具体决策过程

**根据岗位类型调整筛选侧重：**
- 产品经理 → 优先选含「需求挖掘 / 用户研究 / 产品决策取舍」的经历
- AI 战略 → 优先选含「竞品分析 / 行业 mapping / 战略建议」的经历
- AI 产品运营 → 优先选含「数据驱动增长 / 用户生命周期 / 活动策划」的经历

每段格式：

---

**经历 N：[经历名称]（公司 + 角色）**

> **为什么选它：** [与 JD 哪条核心要求直接对应，一句话说明]

| 维度 | 内容 |
|------|------|
| **S 背景** | 当时的情况/场景，要具体（时间、规模、背景） |
| **T 任务** | 你具体负责什么，有什么约束条件 |
| **A 行动** | 你做了什么，为什么这样做而不是别的方式 |
| **R 结果** | 量化结果（用 cv.md 真实数字）或具体产出 |
| **复盘** | 如果重来，什么地方会做不同？这个经历让你明白了什么？ |

---

STAR 拆解结果保存在本次 interview-prep 报告内，供面试复习使用。

```

      ↓                      ↓
      缺口信息              缺口信息
      ↓                      ↓
JD + 评估报告          板块 3（薄弱点追问）
      ↓                板块 4（决策深挖）
模块 1：WebSearch 实时调研
      ↓
  ┌──────────────────────────────────────────────┐
  │  公司重点产品线/近期动态 → 板块 2（趋势题）       │
  │                          板块 4（类比题）        │ ← cv.md（经历细节）
  │  _profile.md 立场观点   → 板块 1（底层认知）      │
  │                          板块 2（趋势判断）       │
  └──────────────────────────────────────────────┘
      ↓
模块 2：问题预测（四板块，12-20 题）
      ↓

  gap-roadmap 🔥   → P0/P1 未完成缺口行动
      ↓

```

## 保存输出

完整报告保存至 `interview-prep/{###}-{公司名拼音或英文}-{YYYY-MM-DD}.md`

- `{###}` = 对应 JDlist 中该 JD 的编号；若本次未关联已有 JD，先执行 JD 协议（见 `_shared.md`）

报告头部格式：

```markdown
# 面试备战：{公司} — {岗位}

**JD 编号：** #{###}
**关联快速评估：** [brief/{编号}-{...}.md]
**备战日期：** {YYYY-MM-DD}
**信息来源：** WebSearch 实时搜索（{日期}）
```

## 铁律

1. **不做 interview-strategy 的工作：** 自我介绍话术、展示页面、现场答题技巧——一律不做，那是另一个 mode 的职责
2. **公司调研必须 WebSearch：** 训练数据里的公司信息可能过时，模块 1 的所有内容必须来自实时搜索
3. **四个模块必须互相联动：** 前置摘要的缺口 → 板块 3+4；模块 1 的发现 → 板块 2+4；板块 3 的薄弱点 → 模块 3 的 P0/P1 行动——如果跨模块没有交叉引用，输出不合格
4. **问题必须个性化：** 每道题都要能说出"为什么针对这个用户问这道题"，否则删掉
5. **答题提示严守真实性：** 只调用用户真实经历和观点，不给没有事实来源的立场——如果某个观点用户在 cv.md 或 _profile.md 里没有留下痕迹，提示框架而非捏造内容
6. **搜不到就如实说：** 某条信息搜不到，写"未找到公开信息"，不用训练数据填充
7. **前置摘要只引用不重新生成：** 四象限表直接引用 gap-roadmap 产出，interview-prep 不重新计算或修改缺口评级

# Mode: interview-review — 面试复盘

用户面试结束后，粘贴文字稿，AI 进行结构化复盘，产出「表现诊断 + 下次怎么改」报告，并将新发现的缺口和问题类型回流到系统。

**分工边界：** interview-prep 解决"面试前准备什么"，interview-review 解决"面试后哪里做得好/不好、下次怎么改"。不重复做问题预测或备考清单。

---

## 输入

用户必须提供：
1. **面试文字稿**（用户已自行从录音转写）
2. **关联的 brief 快速评估文件名或 JD 编号**（如 `001-tal-education-2026-06-02` 或 `#001`，兼容旧 `reports/` 目录）
3. **面试轮次**（HR / 业务 / 技术 / 终面）

**发言人区分：** 根据文字稿中的发言人标注（如"面试官：""我："）区分双方对话。如果用户没有标注发言人，在执行任何分析前主动询问：

> "我看到文字稿里没有区分发言人。请告诉我：面试官的发言是用什么标记的？你自己的回答是用什么标记的？或者你可以重新粘贴一份标注好的版本。"

**轮次缺失：** 如果用户没说明轮次，在执行分析前询问：

> "这是哪一轮面试？（HR / 业务 / 技术 / 终面）轮次会影响我对'这个问题回答得是否合适'的判断。"

**不要凭猜测分析。** 以上信息缺失时，等用户补充后再继续。

---

## 依赖检查

运行前检查以下文件：

**强依赖（缺失则停下）：**
- 评估报告（`brief/{###}-*.md` 或旧 `reports/{###}-*.md`）— 用于 JD 核心要求对照与 gap 比对

若评估报告不存在：
> ⛔ 未找到关联的 brief 报告。interview-review 需要对照 JD 要求和 gap 分析才能完成匹配度评估。请提供 brief 文件名或 JD 编号。

**弱依赖（缺失则提示，降级继续）：**
- `interview-prep/{###}-*.md` — 原定备考计划（用于对比"准备了什么"vs"实际遇到了什么"）

若 interview-prep 不存在：
> ⚠️ 未找到 `interview-prep/{###}-*.md`，无法对比备考计划与实际面试，将跳过备考覆盖率分析。

---

## 加载关联文件

收到有效输入后，必须读取：
- 评估报告：优先读 `brief/{用户提供的文件名}.md`，找不到则从 `reports/{文件名}.md` 读（旧格式兼容）— 获取 JD 要求、评分、差距分析
- `cv.md` — 核对用户面试中的表述是否与真实经历一致
- `modes/_profile.md` — 求职策略上下文（目标 archetype、卖点框架、面试话术倾向）
- `interview-prep/{相关文件名}.md`（如存在）— 了解用户的原定备考计划

---

## 文字稿质量预检

在正式分析前，快速扫描文字稿质量，判断是否满足最低分析条件：

**可分析的最低条件：**
- 能区分面试官和候选人的发言
- 候选人回答总字数不少于 300 字
- 能识别至少 3 个明确的问答对

**如果不满足：** 告知用户具体缺什么，提供修复建议（见"文字稿质量差的处理"章节），**不要强行分析**。

**如果满足但有局部问题**（部分片段乱码、部分发言不清晰）：标注哪些片段无法分析，对可分析部分正常进行，在报告开头注明"以下分析基于可识别片段，XX 处因转写质量问题跳过"。

---

## 五维度复盘

**铁律：每个评价必须引用原文片段。** 不得出现"你的沟通可以更清晰"这类没有原文依据的空话。格式固定为：

> 在回答「[面试官问题]」时，你说「[原话片段，不超过 50 字]」，[具体问题描述]。

---

### 维度 1 — 回答结构

评估每个主要问题的回答是否有清晰的叙述结构（STAR、金字塔、问题-方案-结果等），还是想到哪说到哪。

输出格式：

```
【结构评估】

✅ 结构清晰的回答：
- 「[问题]」→ 使用了 [何种结构]，表现：[一句话说明]

⚠️ 结构待改进的回答：
- 「[问题]」→ 你说「[原话片段]」
  问题：[缺少了什么结构要素，例：没有交代背景/行动跳跃到结果]
  建议：[一句话说怎么改]
```

---

### 维度 2 — 内容质量与真实性

核对回答中的具体事实：数字、时间、项目名称、职责描述是否与 `cv.md` 一致。

重点检查：
- 是否出现夸大（说了比 cv.md 更高的数字或更大的职责范围）
- 是否模糊化（把具体的事说得很笼统，导致无法追问）
- 是否说不出细节（当面试官追问时，回答变得含混）

输出格式：

```
【内容核查】

✅ 与 cv.md 一致、细节扎实：
- [描述哪条回答表现好，引用原话]

⚠️ 需要注意：
- 你说「[原话片段]」
  核查：cv.md 中对应的是「[cv.md 原文]」
  风险：[夸大 / 模糊 / 细节不足]，追问时可能暴露

❌ 与 cv.md 不符：（如有）
- 你说「[原话片段]」，但 cv.md 中写的是「[原文]」
  建议：面试现场如果可以，补充说"准确来说是……"
```

---

### 维度 3 — 岗位匹配度

对照评估报告（`brief/` 或旧 `reports/` 目录）中的 JD 核心要求，检查哪些关键点被主动提及、哪些被遗漏。

输出格式：

```
【JD 关键点覆盖】

✅ 提到了：
- [JD 要求1]：在回答「[问题]」时提及，原话：「[片段]」

❌ 遗漏了：
- [JD 要求2]：整场面试没有主动提及
  本可以在「[哪个问题]」里嵌入，建议话术方向：[一句话]

- [JD 要求3]：同上
```

---

### 维度 4 — 暴露的能力缺口

识别回答中明显薄弱的地方：用户停顿、跑题、答非所问、主动承认不会、面试官重复追问同一问题等。

对每个缺口，标注：
- 严重程度：**高**（直接影响录用判断）/ **中**（有减分但可补救）/ **低**（细节瑕疵）
- 是否新发现：对比评估报告（`brief/` 或旧 `reports/` 目录）的 B 模块差距分析，判断是已知缺口还是新出现的

输出格式：

```
【能力缺口】

[严重程度] [已知 / 🆕 新发现] [缺口名称]
- 触发问题：「[面试官问题]」
- 表现：你说「[原话片段]」，[问题描述]
- 回流建议：[见第三部分"回流建议"]
```

---

### 维度 5 — 沟通表达

识别影响表达质量的具体问题：冗长啰嗦、跑题、自我贬低、过度修饰、高频口癖（嗯/那个/可能/应该是吧）等。

输出格式：

```
【沟通表达】

✅ 表达亮点：[如果有，具体引用]

⚠️ 具体问题：
- [问题类型]：你在「[问题]」的回答中说「[原话片段]」
  影响：[具体说明这样说的问题]
  改法：[一句话建议]
```

---

## 产出结构

### 一、整体诊断（一句话）

```
这场 [轮次] 面试的整体表现：[一句话，直接判断，不含模糊词]
```

不写"表现不错但还有提升空间"这类废话。写"结构完整但岗位关键词覆盖率低，面试官核心关切（AI产品化经验）未被主动提及"这样的判断。

---

### 二、五维度详细复盘

按维度 1-5 顺序输出，每维度引用原文（格式见上方各维度规范）。

---

### 三、回流建议

这部分决定复盘报告是否真正改变下一次面试的准备。

**3-A 新发现的能力缺口 → gap-roadmap**

列出本次面试新暴露的（在评估报告 B 模块差距分析中未曾出现的）能力缺口，在报告内按以下格式展示：

```
建议加入 gap-roadmap：
- 缺口：[描述]
- 影响度：高 / 中 / 低
- 补齐成本：高 / 中 / 低
- 四象限：🔥 / 📅 / 🆓
- 建议行动：[具体可执行的第一步]
```

> **执行方式：** 将以上条目追加写入 `data/gap-backlog.md`（staging 文件）。若文件不存在则创建，并写入以下初始结构：
>
> ```markdown
> # Gap Backlog
>
> <!-- interview-review 发现的新缺口暂存区。gap-roadmap 启动时会读取此文件。 -->
>
> ## 已归档
> ```
>
> 每次追加时，在 `## 已归档` 标题**之前**插入新条目，格式如下（条目间用 `---` 分隔）：
>
> ```markdown
> ### [{公司} — {岗位} | {轮次}面 | {YYYY-MM-DD}]
> **关联报告：** [brief/{文件名}]（或旧 reports/ 目录）
>
> - 缺口：[描述]
> - 影响度：高 / 中 / 低
> - 补齐成本：高 / 中 / 低
> - 四象限：🔥 / 📅 / 🆓
> - 建议行动：[具体可执行的第一步]
>
> ---
> ```
>
> 用户下次运行 `/nogap gap-roadmap` 时，该文件会被提示是否并入主分析。不直接修改用户已确认过的 gap-roadmap 输出。

**3-B 没准备好的问题类型 → interview-prep**

列出本次暴露的、下次同类岗位面试前应该准备的问题类型：

```
建议加入面试准备 [P0/P1]：
- 问题类型：[描述]
- 触发场景：[这次被问时出了什么问题]
- 建议准备方向：[1-2 句]
```

> **执行方式：** 若 `interview-prep/` 目录下存在同岗位的 interview-prep 文件，将以上条目追加到该文件末尾的「复盘补充」区块。若文件不存在，输出给用户手动处理。

**3-C 面试中提到但 cv.md 未体现的真实经历**

如果用户在面试中提到了真实发生过但 cv.md 里没有记录的经历/成果：

```
建议补入 cv.md：
- 经历描述：[用户说了什么]
- 建议写法：[符合 cv.md 格式的一条表述]
- 注意：补入前请用户自行确认细节准确，AI 不直接修改 cv.md
```

> **执行方式：** 只输出建议，**不自动修改 cv.md**。cv.md 是唯一事实源，任何修改须由用户主动操作。

---

### 四、下次面试 3 个最重要的改进点

按优先级排列，每条必须具体可执行，不写"加强结构感"这类空话：

```
**改进点 1（最重要）：**
[具体行动，例："在讲经历前先说一句定位句，格式：'这件事发生在[时间/背景]，我负责的是[具体事]'"]

**改进点 2：**
[...]

**改进点 3：**
[...]
```

## 文字稿质量差的处理

**情形 1：没有发言人区分**
→ 停止分析，询问用户补充标注。不要猜测哪句话是面试官说的、哪句是用户说的。

**情形 2：部分片段乱码或无法识别**
→ 在报告开头标注"[注：以下片段因转写质量问题跳过分析：行 X-Y, 行 Z-W]"，对可识别部分正常分析，降低对跳过片段的依赖。

**情形 3：内容太少（少于 300 字有效候选人回答）**
→ 告知用户："当前文字稿中可识别的候选人回答约 [N] 字，不足以支撑五维度分析。建议补充完整转写稿，或告知是否只需要对现有片段做部分分析。"

**情形 4：大量转写错误但整体可理解**
→ 正常分析，在引用原话时标注 `[转写可能有误]`，不因错别字做内容层面的负面评价。

## 保存输出

报告保存至 `interview-review/{###}-{公司名拼音或英文}-{YYYY-MM-DD}.md`

- `{###}` = 对应 JDlist 中该 JD 的编号（从关联的评估报告文件名读取）

报告头部：

```markdown
# 面试复盘：{公司} — {岗位} | {轮次}

**JD 编号：** #{###}
**面试日期：** {用户提供或从文字稿推断}
**关联评估：** [brief/{文件名}.md]（或旧 reports/ 目录）
**复盘日期：** {YYYY-MM-DD}
**文字稿质量：** 完整 / 部分（注明跳过片段）
```

---

## 前向指针

复盘报告完成并保存后，在对话末尾输出：

```
✅ 面试复盘完成 → interview-review/{###}-{公司}-{日期}.md
```

若有新缺口写入 gap-backlog，追加：
```
📥 {N} 个新缺口已写入 gap-backlog。
   下次投新 JD 后运行 gap-roadmap 时，系统会提示是否并入分析。
```

---

## 铁律

1. **引用原文，不说空话：** 每个评价必须指向文字稿中的具体片段，格式"你说「…」，问题是…"
2. **诚实但不打击：** 指出问题要具体，措辞要建设性。不写"你这个回答完全没有价值"，写"这个回答缺少了…，如果加上…会更有说服力"
3. **真实性双向：** 不凭空指责用户没说过的话，也不因用户说得笼统就脑补细节
4. **全程中文**
5. **回流必须执行：** 3-A 的新缺口必须写入 `data/gap-backlog.md`；3-B 的问题类型如果有对应 interview-prep 文件必须追加进去；3-C 只输出建议，不改 cv.md

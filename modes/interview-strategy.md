# Mode: interview-strategy — 面试现场展示工具

服务「面试进行中」这个时刻。产出两样面试现场真的会用的东西：

1. **针对岗位的 100 字自我介绍**（开场用，口语化、可背诵）
2. **可共享屏幕的个人展示网页**（整场面试跟着讲，生成 HTML 文件）

**和 interview-prep 的分工：** interview-prep 解决"面试前准备什么"，interview-strategy 解决"面试中怎么讲、怎么展示自己"。不重复做问题预测、备考清单。

---

## 依赖检查

运行前检查以下文件：

**强依赖（缺失则停下）：**
- `JDlist/{###}.md` — JD 原文与系统分析结论

若"系统分析结论"区块为占位符（brief 未运行）：
> ⛔ 未找到 `brief/{###}-*.md`。interview-strategy 需要 brief 写入的人才画像结论来决定展示哪些经历。请先运行 `/nogap brief #{###}`。

**弱依赖（缺失则提示，降级继续）：**
- `cv-tailor/{###}-*.md` — 改写后的经历表述（可提升自我介绍质量）

若 cv-tailor 不存在：
> ⚠️ 未找到 `cv-tailor/{###}-*.md`，自我介绍将直接基于 cv.md 原始表述。建议先运行 `/nogap cv-tailor #{###}` 获得更精准的经历包装。

## 输入

运行前必须读取：
- `JDlist/{###}.md` → 读取"JD 原文"和"系统分析结论"（Archetype + 人才画像摘要，直接复用，不重新分析 JD）
- `cv.md` — 用户真实经历，唯一事实源
- `modes/_profile.md` — 求职策略上下文（目标 archetype、卖点框架、面试话术倾向）
- `config/profile.yml` — 读取求职阶段
- `cv-tailor/{###}-*.md`（如存在）— 读取改写后的经历表述用于自我介绍

## 网页生成前确认（强制）

`interview-strategy` 采用两阶段流程。**用户确认网页配置前，不生成 HTML。**

### 第一阶段：收集网页配置

如果用户没有在指令中给出完整配置，先结合 JD 和 `cv.md` 推荐一套默认值，再一次性询问以下项目，并提醒用户补充本轮面试需要展示的项目材料：

1. **本轮想重点提到的过往经历**
   - 从 `cv.md` 中推荐 3-5 段与 JD 最相关的经历 / 项目
   - 用户可以选择推荐项，也可以指定其他经历
   - 用户明确指定的经历优先进入网页，除非与 JD 完全无关或有隐私风险
2. **个人主题色**
   - 询问用户是否有个人主题色 / 品牌色
   - 如果用户说有，并给出颜色名、色系或 HEX 色值，把这一信息写入 `modes/_profile.md` 的“视觉偏好”区
   - 如果用户说没有，或没有提供可执行的颜色信息，则按照 JD 气质推荐主题色
3. **本轮网页主题色**
   - 靛青：理性、专业，适合战略 / 研究 / 投资
   - 科技蓝：清晰、现代，适合 AI / 产品 / 技术
   - 墨绿：沉稳、可信，适合商业 / 咨询 / 运营
   - 酒红：成熟、有辨识度，适合品牌 / 内容 / 创意
   - 石墨：克制、中性，适合希望弱化视觉风格的场景
   - 用户也可以提供自定义 HEX 色值；若已有个人主题色，默认优先使用个人主题色
4. **保留板块**（多选）
   - 自我定位
   - 核心数字
   - 相关经历
   - 岗位适配
   - 个人项目 / 研究
   - 加入团队后的贡献
   - 自我优势
   - 职业规划
5. **内容颗粒度**
   - 精简：每项 1 句话，只作为讲述提词与视觉证据
   - 标准：每项 1-2 句话，兼顾现场讲解和面试官自行浏览
   - 详细：保留方法、过程和结果，并引用简历中的关键事实，但仍不写成完整讲稿
   - 默认推荐使用“标准偏详细”：因为面试官往往不会仔细阅读简历，网页需要二次强调核心经历
6. **视觉风格**
   - 专业清晰：强层级、稳定网格，默认推荐
   - 数据报告：突出指标、方法和结论
   - 产品编辑：更强调项目、作品和叙事节奏
7. **预计讲解时长**
   - 3 分钟：3-4 个板块
   - 5 分钟：4-6 个板块，默认推荐
   - 10 分钟：6-8 个板块，可保留更多案例
8. **项目材料（可选）**
   - 公开链接：项目网站、GitHub、在线报告、作品集页面
   - 本地文件：PDF 报告、项目截图、HTML Demo、Markdown 文档、演示文件
   - 每份材料最好附上：标题、它证明什么、是否允许面试官打开或下载
   - 如果本轮没有需要展示的材料，回复“无项目材料”即可

询问格式：

```text
生成网页前，请确认配置（可直接回复“按推荐生成”）：

1. 本轮想重点提到的经历：[基于 JD 推荐 3-5 段]
2. 你有自己的个人主题色吗？如果有，请给颜色名或 HEX 色值；如果没有，我会按 JD 推荐
3. 本轮网页主题色：[基于个人主题色或 JD 的推荐值]
4. 保留板块：[基于 JD 推荐的板块]
5. 颗粒度：精简 / 标准 / 详细（推荐：标准偏详细）
6. 视觉风格：专业清晰 / 数据报告 / 产品编辑
7. 讲解时长：3 / 5 / 10 分钟

项目材料（可选）：
- 请粘贴公开链接，或上传 / 提供本地文件路径
- 请注明每份材料的标题、它证明什么、是否允许面试官打开或下载
- 没有则回复“无项目材料”

隐私确认（可选）：是否有不希望网页出现的信息？
```

### 配置默认值

用户回复“按推荐生成”时，直接使用系统给出的推荐值。用户只回答部分项目时：

- 已回答的项目严格采用用户选择
- 未回答项目采用本轮推荐值，不重复追问
- 用户指定的重点经历优先进入网页，并在报告中记录
- 用户提供个人主题色时，先写入 `modes/_profile.md`，再用于本轮网页；用户没有个人主题色时，不写入 profile
- 默认不展示电话、邮箱、住址、照片等个人敏感信息
- 只有 `cv.md` 或用户消息中明确提供的公开链接才允许放入网页
- 用户未提供项目材料时，不生成空的项目入口或“待上传”占位符

### 第二阶段：生成成品

收到配置后再生成报告和 HTML，并把最终配置写入报告。**配置是生成参数，不是网页内容。**

若用户提供个人主题色，更新 `modes/_profile.md`，建议格式：

```markdown
## 视觉偏好

- 个人主题色：#4F46E5（靛青）
- 使用规则：interview-strategy 默认优先采用个人主题色；如与 JD 气质明显冲突，再在生成前询问是否切换
```

### 项目材料处理

对用户提供的材料逐项执行：

1. **确认用途与权限**
   - 只收录与本轮 JD 和讲述策略相关的材料
   - 用户未明确允许公开或打开的材料，不放入成品网页
   - 检查文件中是否包含电话、邮箱、住址、客户机密、内部数据等敏感信息；存在风险时先提示用户
2. **公开链接**
   - 保留原始 URL，生成清晰的项目卡片和操作文案，如“查看项目”“打开报告”
   - 不把裸 URL 直接堆在页面正文中
3. **本地文件**
   - 验证文件存在后，复制到 `output/assets/{###}/`
   - 使用小写英文、数字和连字符重命名，避免空格、中文路径和原始绝对路径泄露
   - HTML 只写相对路径，如 `assets/000/market-research.pdf`
4. **展示方式**
   - 图片：可作为项目卡片缩略图，点击查看原图
   - PDF / HTML：点击在新标签页打开
   - Markdown：转换为 HTML 摘要页，或仅作为下载材料
   - PPTX / DOCX / XLSX：提供明确的“打开文件”或“下载文件”链接，不伪装成网页预览
5. **可移植性**
   - 没有本地材料时，网页仍保持单 HTML 文件
   - 有本地材料时，HTML 与 `output/assets/{###}/` 共同组成可分享文件包；不得引用用户电脑上的绝对路径

## 产出一：100 字针对岗位自我介绍

### 要求

- **严格控制在 100 字左右**（中文计字，±10 字可接受）
- **口语化**：是"能直接对着面试官背出来"的话，不是书面简历语言。每句话说出来都要自然，不出现"在此过程中""深度参与""赋能"等书面词
- **结构**：一句话定位（我是谁）→ 1-2 个最相关的经历亮点 → 为什么适合这个岗位
- **踩中 JD 最看重的 2-3 个点**（从 JD 人才画像提取，不超过 3 个，宁少勿滥）
- **所有内容基于 cv.md 真实经历**，不编造、不添加 cv.md 里没有的事实

### 输出格式

先给出自我介绍正文，然后在下方注明：

```
【字数】XX 字
【踩中的 JD 关键点】
- [点1]：[介绍里哪句话体现了这个点]
- [点2]：[介绍里哪句话体现了这个点]
- [点3（如有）]：[...]
【事实依据】所有内容均来自 cv.md，逐条列出对应来源
```

## 产出二：可共享屏幕的面试展示网页

### 内容筛选原则

复用 JD 人才画像的分析结果，并根据用户选择的板块、颗粒度和讲解时长筛选内容：

- **必放**：用户选择的重点经历，以及与 JD 核心要求直接对应的 1-2 段工作/实习经历
- **优先放**：与 JD 高度相关的项目经历（如 NOgap 对 AI PM 类岗位）
- **材料优先级**：用户上传且与 JD 高度相关的项目材料，优先于只有文字描述、没有证据入口的次要项目
- **选择性放**：最相关的能力证明（关键数字、工具、方法）
- **不放**：与 JD 关联度低的经历（即使简历里有）、完整教育背景（只放最关键的 1 行）、通用技能列表

网页从上到下的顺序 = 用户自我介绍的讲述顺序，让用户能顺着往下讲。

### 经历内容深度

面试展示页不是普通个人主页。默认要比作品集更贴近简历事实，因为面试官往往不会细读简历，网页需要帮助用户二次强调核心经验。

每段重点经历建议包含：

- **场景 / 任务**：为什么做这件事，解决什么问题
- **行动 / 方法**：用户具体做了什么，使用了什么框架、工具或流程
- **结果 / 指标**：来自 `cv.md` 的数字、交付物或上线结果
- **与 JD 的关系**：这段经历证明了岗位要求中的哪项能力

禁止把经历写成大段简历复刻。优先使用短段落 + 2-4 个事实点 / 标签 / 指标卡。

颗粒度约束：

| 颗粒度 | 单个内容项 | 页面定位 |
|--------|------------|----------|
| 精简 | 标题 + 1 句证据，建议不超过 35 字 | 现场提词与视觉辅助 |
| 标准 | 标题 + 1-2 句证据 + 方法标签 / 指标 | 现场讲解 + 会后浏览 |
| 详细 | 标题 + 场景 / 行动 / 结果 / JD 关系，不超过 4 个短信息块 | 深度案例展示 |

### 视觉设计规范

- **布局**：默认使用左侧固定导航 + 右侧内容模块；若用户选择的视觉风格需要变化，可以调整内容网格，但必须保持单页和清晰层级
- **关键数字突出**：凡是量化数字（如 12%→35%、26→5、CPM $1.0、AUC 0.82）用加粗或高亮色展示，让面试官一眼能看到证明点
- **留白充分**：每个模块之间有足够间距，不堆砌，面试官能在 10 秒内扫描完一屏内容
- **配色**：严格采用用户确认的主题色；确保正文对比度和投屏可读性，主题色只用于重点、链接和状态
- **不要**：动效、JS 动画、外部字体 CDN（离线可用）、广告感装饰、照片

### 图片与流程图展示规范

如果网页中加入流程图、截图、架构图或项目材料图片：

- 关键图（如系统架构、流程图、核心产品截图）应使用**大图展示区**，不要压成小缩略图
- 桌面端大图区域建议占内容宽度的 60%-100%，高度不低于 360px；复杂图可单独占一整行
- 移动端图片宽度 100%，允许纵向滚动，不横向溢出
- 图片必须配一句短 caption，说明“这张图证明什么”
- 点击图片应能在新标签页打开原图或对应材料
- 若图片信息密度高，优先使用 SVG / PNG 原图，不用低清截图

### 成品页面纯净原则

生成后的 HTML 是可以直接用于面试录制、屏幕共享或发送给面试官的成品，页面中不得出现：

- 主题色选择器、风格选择器、板块开关、颗粒度设置
- “修改网页”“继续生成”“由 NOgap 生成”等制作说明
- 面向候选人的操作教程、设计解释或配置参数
- 调试信息、占位文字、未完成链接
- “请上传项目”“暂无材料”等面向生成阶段的提示

主题色和风格必须在生成时固化到 CSS。除页面导航和项目链接外，不添加与面试展示无关的交互。

### 技术规范

- 生成**单个独立 HTML 文件**，内嵌所有 CSS（`<style>` 标签），无外部依赖
- 用户提供本地项目材料时，允许同时生成 `output/assets/{###}/`；HTML 必须使用相对路径引用
- 用本地安全字体：`font-family: -apple-system, BlinkMacSystemFont, "Segoe UI", "PingFang SC", "Microsoft YaHei", sans-serif`
- 文件名格式：`output/{公司名拼音或英文}-{岗位关键词}-showcase-{YYYY-MM-DD}.html`
- 确保在本地浏览器直接打开（`file://`）可正常显示，不依赖服务器

### HTML 结构模板

```html
<!DOCTYPE html>
<html lang="zh-CN">
<head>
  <meta charset="UTF-8">
  <meta name="viewport" content="width=device-width, initial-scale=1.0">
  <title>{姓名} — {岗位}</title>
  <style>
    /* 内嵌所有样式，无外部依赖 */
    /* 配色：深灰文字 + 白底 + 蓝灰点缀 */
    /* 字体：本地安全字体栈 */
  </style>
</head>
<body>
  <!-- Hero 区：姓名 + 一句话定位 + 最核心的 2-3 个数字亮点 -->
  <!-- 模块 1：与 JD 最相关的工作/实习经历 -->
  <!-- 模块 2：与 JD 最相关的项目经历 -->
  <!-- 模块 3（可选）：关键能力证明 / 技术栈 -->
  <!-- 模块 4（可选）：教育背景（1 行）+ 语言 -->
</body>
</html>
```

Hero 区必须包含：
- 姓名 + 「→」+ 目标岗位（如：杨思嫣 → AI产品经理实习）
- 一句话定位（与自我介绍第一句保持一致）
- 用户保留“核心数字”板块时，展示 2-4 个最关键的量化亮点

## 前向指针

展示方案完成并保存后，在对话末尾输出：

```
✅ 面试展示方案完成
   → interview-strategy/{###}-{公司}-{日期}.md（自我介绍 + 方案说明）
   → output/{公司}-{岗位关键词}-showcase-{日期}.html（展示网页）

面试结束后复盘：/nogap interview-review #{###}
```

## 铁律

1. **不重复 interview-prep 的工作**：不做问题预测、不做备考清单、不做公司调研——那是 interview-prep 的职责
2. **真实性**：自我介绍和网页内容全部基于 cv.md 真实经历，数字必须是简历里真实出现的数字，不编造
3. **针对性**：同一个用户面试不同岗位，生成的自我介绍和网页应该不同——内容由 JD 人才画像决定，不是通用简历的搬运
4. **可用性**：HTML 必须能直接用浏览器打开，不依赖外部服务、不需要联网
5. **先确认后生成**：网页配置不完整时必须先询问；用户确认前不得生成 HTML
6. **成品无制作控件**：所有配置在生成时固化，HTML 中不得保留生成或编辑界面

## 保存输出

### 文字产出（自我介绍 + 分析）

保存至 `interview-strategy/{###}-{公司名拼音或英文}-{YYYY-MM-DD}.md`

- `{###}` = 对应 JDlist 中该 JD 的编号；若本次未关联已有 JD，先执行 JD 协议（见 `_shared.md`）

报告头部格式：

```markdown
# 面试展示方案：{公司} — {岗位}

**JD 编号：** #{###}
**日期：** {YYYY-MM-DD}
**展示网页：** [output/{公司}-{岗位关键词}-showcase-{YYYY-MM-DD}.html]
```

### 展示网页（HTML）

继续保存至 `output/{公司名拼音或英文}-{岗位关键词}-showcase-{YYYY-MM-DD}.html`

```
/nogap interview-strategy
```
